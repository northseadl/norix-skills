# Product Anchor — 产品边界锚定协议

## 这个协议解决什么问题

AI Agent 面对"不确定怎么做"的功能时，会选择**看起来完成了**的实现方式——占位函数、硬编码数据、只验证渲染不验证数据流的测试。这不是 Agent 偷懒，而是**信息不足时的理性选择**：与其实现一个可能错误的逻辑，不如放一个占位符等待更多信息。

问题在于：这些占位符会伪装成已完成的工作，在代码审查和测试中幸存，逐渐累积为系统性技术债。

**产品边界文档 (PBD)** 的目标不是约束 Agent，而是**消除信息不足**——为每个功能提供足够精确的行为定义，让 Agent 没有理由选择占位实现，因为它已经知道了"怎么做"。

## 核心思维模型：因果链

PBD 的根基不是文档格式，而是一种思维方式——**因果链追踪 (Causal Chain Tracing)**。

每个用户可见的数据和交互，都存在一条从源头到渲染的因果链：

```
数据因果链:
  数据库表/外部API → 后端路由处理 → API 响应 JSON → 前端请求 → 状态管理 → UI 渲染

交互因果链:
  用户操作 → 事件处理器 → 业务逻辑 → 状态变更/导航/API调用 → UI 反馈
```

**占位实现的本质是因果链断裂**：
- 硬编码数据：链条在"前端请求"处断裂，数据不来自 API
- notImplemented：链条在"事件处理器"处断裂，操作不触发业务逻辑
- Testing Theater：测试验证的是断裂后的假链条，不是真实因果链

所以，定义产品边界的核心工作不是填表，而是**为每个功能绘制完整的因果链，让每一环都可追溯、可验证**。

## PBD 在知识体系中的角色

```
soul.md     → 怎么思考（跨项目通用原则）
domain.md   → 怎么构建（技术架构与选型真相）
PRODUCT.md  → 做什么 & 做到什么程度（产品行为的因果链定义）
```

domain.md 说"用什么工具"，PRODUCT.md 说"用这些工具要达到什么效果，效果如何验证"。

## 功能的完整性证明

一个功能"真正实现了"意味着什么？不是"页面能渲染"，而是以下三个条件同时满足：

### 1. 数据因果链完整

从源头到渲染，每一环都存在且可独立验证。关键问题：
- 数据从哪个表/API来？
- 中间经过什么变换？
- 前端通过什么方式获取？
- 渲染用了响应中的哪个字段？
- **如果修改源头数据，渲染结果是否随之改变？**（这是终极验证）

### 2. 状态空间完整

因果链不是只有一条——不同条件下链条走向不同。至少存在这些分支：
- 正常路径：数据存在且请求成功
- 加载路径：请求已发出但未返回
- 空数据路径：请求成功但结果为空
- 错误路径：请求失败或数据异常

每条路径都需要独立的 UI 表现和用户可执行的操作。如果只有正常路径有 UI，其余状态要么是空白、要么静默降级到 mock 数据——这就是因果链不完整。

### 3. 交互因果链闭环

每个用户可操作的元素都应有明确的因果链：
- 操作触发什么？
- 触发后数据流向哪里？
- 最终产生什么可观测的结果？

如果一个按钮的处理器是 `toast("即将开放")`，那因果链在事件处理器处断裂。

## 如何描述一个功能

不需要死板的模板。每个功能的描述应该让一个对项目一无所知的 Agent 读完后能够：
1. 知道这个功能**是什么、从哪进入**
2. 追踪每一条**数据因果链**（从源头到渲染）
3. 追踪每一条**交互因果链**（从操作到结果）
4. 理解所有的**状态分支**和对应的 UI 表现
5. 用具体的验证方法确认实现是**真实的而非伪装的**

以下维度是实现上述目标的自然结果，不是需要死记的清单：

**身份与入口** — 功能定位、导航路径、路由参数。让 Agent 知道"在哪"。

**数据因果链** — 每条数据从源头到渲染的完整路径。这是防伪的第一道防线——声明了数据来源，硬编码就无处藏身。关键是写清楚：来源端点、响应字段、前端如何获取和使用。刷新和分页策略也属于数据因果链的一部分。

**交互因果链** — 用户的每个操作触发什么、数据流向哪、产生什么结果。不是"支持点击"，而是"点击后发生了什么完整链条"。

**状态空间** — 除了正常展示数据外，还有哪些状态？加载中、空数据、错误、加载更多……每种状态的 UI 和可执行操作是什么？这迫使思考因果链的所有分支路径。

**验收标准** — 如何证明上面描述的因果链是真实存在的而非伪装的？每条标准都应该是一个**可操作的实验**："做 X → 观察 Y"。如果一个占位实现也能通过这个实验，标准就是无效的。

### 验收标准的思维方式

好的验收标准不是检查清单，而是**因果链的可观测实验**：

```
弱: "页面显示价格"
  → 硬编码 <Text>7.2345</Text> 就能通过

强: "页面价格来自 GET /api/market/:symbol 的 price 字段；
    后端 market_quotes 表中该 symbol 的 price 更新后，
    前端刷新显示新值"
  → 这是一个因果链实验：修改源头 → 中间传导 → 末端变化
```

几种有效的验证模式：
- **源头变更传导**：修改上游数据 → 下游同步变化（验证数据因果链完整）
- **链路追踪**：操作 A → 触发 B → 到达 C 且携带正确参数（验证交互因果链完整）
- **分支覆盖**：构造特定条件（断网/空数据/越界）→ 系统按预期响应（验证状态空间完整）
- **否定性实验**：生产构建中搜索不应存在的模式（占位符/mock引用/硬编码数据）

## 四种工作流

### Genesis — 从代码逆向提取产品边界

项目尚无 PBD 时，从代码现实出发——因为**代码是唯一的事实**。

思维过程：
1. 从路由配置和页面目录建立功能清单
2. 对每个功能，追踪其数据因果链——数据是来自真实 API 调用还是 hardcoded/mock？链条在哪一环是真实的，在哪一环断裂？
3. 追踪交互因果链——事件处理器执行了真实逻辑还是占位？
4. 检查状态空间——是只有 happy path 还是覆盖了加载/错误/空数据？
5. 基于以上分析，标记每个功能的真实状态：
   - 因果链完整 + 状态空间覆盖 → **DONE**
   - 有页面但因果链断裂（hardcoded/mock/占位）→ **STUB**
   - 规划中未实现 → **SPEC**
6. 对 DONE 功能写出完整因果链描述；对 STUB 功能标注断裂点和缺失维度

### Expansion — 为新功能定义产品边界

用户描述新功能时，核心工作是**补全 Agent 缺失的因果链信息**。

思维过程：
1. 从用户描述中识别涉及的数据和交互
2. 为每条数据设计完整的因果链——从哪来、经过什么、到哪去
3. 为每个用户操作设计交互因果链——触发什么、流向哪、结果是什么
4. 展开状态空间——正常路径之外还有哪些分支
5. 为每条因果链设计可观测的验证实验
6. 分析对现有功能因果链的影响——新功能是否修改了已有功能的数据源、导航目标或共享状态

用户没有提供的信息，基于项目上下文提出最合理的建议，让用户确认。目标是让功能描述精确到"Agent 读完后没有任何理由选择占位实现"。

### Evolution — 需求变更时更新边界

需求变化时，关键是**追踪变更在因果链网络中的传播**。

思维过程：
1. 识别变更修改了哪条因果链的哪一环
2. 沿因果链向上下游传播——上游数据源变了？下游渲染逻辑要改？
3. 检查跨功能的因果链交叉点——共享的 API、组件、状态
4. 更新所有受影响功能的描述和验收标准
5. 被替代的功能标记 CUT，下个版本清除

### Verification — 审计实现是否符合边界

核心是**验证代码中的因果链是否与 PBD 声明的一致**。

思维过程：
1. 读取 PBD 中声明的因果链
2. 在代码中追踪同一条链——每一环是否存在？是否真实？
3. 分类审计结果：
   - **通过**：代码中的因果链与 PBD 声明一致
   - **未通过**：因果链某环缺失或与声明不符（如缺少 error 状态 UI）
   - **伪装实现**（最高优先级）：因果链看起来存在但实际断裂——数据来自 hardcoded 而非 API、事件处理器是 toast 而非真实逻辑、测试验证 mock 数据而非真实行为
4. 伪装实现需要指出断裂点（文件:行号）和应有的完整链路

## PBD 与测试的关系

PBD 验收标准定义**验证什么（WHAT）**，测试定义**怎么验证（HOW）**：

```
PBD 验收标准: "价格来自 API，后端变更后前端同步"
    │
    ├──→ 单元测试: 组件在给定 props.price 时正确渲染
    ├──→ 集成测试: API 返回 {price: 42} → hook 更新 → 组件显示 42
    └──→ E2E 测试: 修改 DB → 刷新页面 → 看到新价格
```

测试如果只验证"组件能渲染某个文本"而不关心文本从哪来，它验证的是一条不存在的因果链——这是 Testing Theater 的根源。好的测试必须能追溯到 PBD 中的某条因果链。

## PBD 如何驱动后续开发

PBD 不是写完就束之高阁的文档。它应该在开发过程中持续发挥作用：

**开发前**：Agent 读取功能的因果链定义，据此实现——而不是猜测。因果链越精确，Agent 选择占位实现的概率越低。

**开发中**：每实现一条因果链就在 PBD 中勾选对应的验收标准。如果发现实际情况与 PBD 描述不符，更新 PBD（代码是事实）。

**开发后**：运行 Verification 工作流，确保所有声明的因果链在代码中真实存在。

**需求变更时**：先更新 PBD（Evolution 工作流），再改代码。这确保变更的影响范围被完整评估后再动手。

## 演进原则

- **PBD 是活文档**：过期的因果链描述比没有描述更危险——它会误导 Agent 实现已经不存在的链路
- **代码是事实**：PBD 与代码冲突时，先确认哪个正确再统一
- **精确优于详尽**：一条精确的因果链描述比十条模糊的需求点更有价值
- **渐进精确**：初版粗略无妨，但每次触碰该功能时必须提升精度
- **删除优于标注**：废弃功能直接移除，Git 记录历史
