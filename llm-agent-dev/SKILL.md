# LLM Agent 设计与开发 — Executable Protocol

> This skill is for designing and evaluating agent systems.
> It is not a repository governance skill and does not define branching policies.

---

## 1) Scope and Boundaries

### In scope

- Agent pattern selection and composition
- Intent and dialogue architecture
- Tool integration contracts
- Guardrails and HITL escalation
- Evaluation strategy and release gating

### Out of scope

- Repository branch governance
- CI/CD platform policy
- Autonomous coding orchestration protocols

### Non-goals

- Do not inject repository governance rules unless explicitly requested.
- Do not produce generic prose without architecture decisions and tradeoffs.

---

## 2) Working Mode

When triggered, execute the following phases in order.

### Phase A — Frame the target

Extract the minimum decision context:

- Business objective (conversion, containment, speed, compliance)
- User channel and latency budget
- Risk tolerance and escalation requirements
- Tool/data boundaries and system constraints

If critical inputs are missing, assume the safest production default and state the assumption.

### Phase B — Select pattern stack

Choose from the 10 patterns below and produce a stack, not a single label.

### Phase C — Produce implementation contracts

Output concrete contracts:

- Intent contract
- State/slot contract
- Tool contract (function or MCP)
- Guardrail contract
- Handoff contract

### Phase D — Produce evaluation plan

Define offline + online metrics, pass/fail thresholds, and rollback triggers.

## 4) Selection Rules

Use these deterministic rules before proposing a design.

1. If latency budget is under 1s, prefer `CREX + Slot` and avoid ReAct.
2. If deterministic tool calls dominate, prefer `Function Calling` first, then `MCP` when scale grows.
3. If steps are policy-fixed, use `Workflow Agent` instead of free-form reasoning.
4. If cross-domain intent ambiguity is high, add `Router-Expert`.
5. If accountability is required, add `Handoff` and explicit HITL checkpoints.
6. Only use `Multi-Agent` when single-agent decomposition fails cost/latency targets.

## 5) Output Contract

Every response generated by this skill must include these sections.

### A. Recommended Stack

- Primary pattern stack (ordered)
- Why this stack fits constraints
- Rejected alternatives and reasons

### B. Runtime Architecture

- Request path (input -> classify -> plan -> act -> respond)
- State model (session, slots, workflow state)
- Tool invocation boundaries

### C. Safety and Escalation

- Guardrail layers (input/action/output)
- Policy violation handling
- HITL handoff triggers

### D. Evaluation Plan

- Offline test set categories
- Online KPIs (containment, success rate, latency, escalation rate)
- Release gate thresholds
- Rollback conditions

If any section is missing, the response is incomplete.

## 8) Delivery Quality Bar

Before finalizing:

- Architecture is internally consistent.
- Latency/cost/safety tradeoffs are explicit.
- Contracts are implementable by engineering teams.
- Evaluation plan is measurable and production-usable.

If these checks fail, revise before returning the answer.
