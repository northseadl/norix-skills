# 状态覆盖规范

## 必备状态

每个数据驱动的页面或组件必须覆盖以下 6 种状态：

| 状态 | 含义 | 优先级 |
|------|------|--------|
| `loading` | 数据获取中 | P0 |
| `empty` | 数据为空 | P0 |
| `error` | 操作或加载失败 | P0 |
| `success` | 操作成功/正常数据展示 | P0 |
| `permission` | 无权限访问 | P1 |
| `partial` | 部分成功/降级展示 | P2 |

## 状态覆盖表模板

每个核心页面都需提供：

| 状态 | 触发条件 | 用户看到的 | 恢复动作 |
|------|---------|-----------|---------|
| loading | 页面初始化/数据请求中 | 骨架屏/加载指示器 | 自动：请求完成后切换 |
| empty | 查询结果为空 | 空状态插画 + 引导文案 + 行动按钮 | 引导用户创建/调整筛选 |
| error | 网络错误/服务异常 | 错误提示 + 影响说明 + 重试按钮 | 点击重试/返回上一页 |
| success | 数据正常返回 | 正常页面内容 | — |
| permission | 用户角色无权限 | 权限说明 + 申请入口 | 联系管理员/申请权限 |
| partial | 部分数据加载失败 | 已加载部分正常展示 + 失败部分标记 | 重试失败部分 |

## 状态设计规范

### Loading 状态
- 首选骨架屏（Skeleton）而非旋转加载器（Spinner）
- 骨架屏布局应与真实内容布局一致
- 短时操作（< 300ms）不显示 loading（避免闪烁）
- 长时操作（> 2s）提供进度估算或步骤说明

### Empty 状态
- 提供上下文相关的引导（不是通用的"暂无数据"）
- 包含明确的行动按钮（"创建第一个 ..."、"调整筛选条件"）
- 首次使用的空状态可包含产品引导

### Error 状态
- 明确说明出了什么问题（用用户能理解的语言）
- 说明影响范围（整页不可用 vs 部分功能受影响）
- 提供恢复动作（重试、返回、联系支持）
- 区分用户可修复（表单错误）和不可修复（服务器故障）

### Permission 状态
- 说明为什么无权限
- 提供获取权限的路径（申请、联系管理员）
- 不暴露敏感页面的存在性（404 vs 403 策略）

## 质量要求

1. 错误态必须给恢复动作，绝不 dead-end
2. 权限态必须给下一步指引
3. success 反馈不得抢占用户流程（toast，非模态弹窗）
4. loading 态必须有超时兜底（> 30s 显示超时提示）
5. 状态之间的切换必须有过渡动画，避免闪烁跳变
