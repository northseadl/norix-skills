# Soul — Agent 技术原则与设计哲学
> 跨项目通用的高维判断力。每条法则都必须通过"删掉它 Agent 会变蠢"的测试。

## 架构原则 (Architectural Principles)
- **Code is Truth**：文档是声明，运行结果才是事实；关键行为必须通过源码或 probe 验证。
- **状态分层**：把"内容状态""视图状态""派生状态"分离，避免一个变量承载多种语义。
- **恢复优先的协作状态**：凡是参与调度、续期、收件箱消费、协作提示的运行时状态，都必须持久化；内存态只能做缓存，不能做团队协作的唯一事实源。
- **骨架与智能分离**：框架负责协议、状态、可恢复协作原语；高判断密度动作（规划、评审、合并策略）默认交给 Agent，而不是写死在框架流程里。
- **运行时产物 Git 中性**：框架生成的 helper/缓存/协作脚本不能污染工作树脏状态；运行时辅助文件要么被忽略，要么落在仓库外。
- **失败显式化**：I/O、网络、导出等边界操作必须校验返回值和状态码，失败必须可见。
- **幂等优先**：刷新、保存、重试、导出、同步等操作应可重复执行且结果可预测。
- **最小依赖面**：能用标准库解决的能力不引入额外依赖，降低维护熵。
- **输出面向消费者**: CLI 默认输出 Agent/人类可直接消费的格式（table/summary），raw JSON 通过 `--format json` 按需提供。
- **凭证与配置内聚**: 凭证/密钥必须存入加密 vault（NX1 等），不允许用环境变量配置。CLI 提供显式的 `setup` 命令完成初始化，而非依赖 export 。
- **追踪优于约定**：管理资源（文件/技能/配置）时，用 manifest/状态文件显式追踪所有权，而非依赖命名约定或白名单。

## 代码哲学 (Code Philosophy)
- **Guard Clause First**：先处理空值、非法输入、前置条件失败，再进入主路径。
- **单一职责函数**：测量、缩放、下载、复制、请求解耦，函数名直接表达意图。
- **错误语义统一**：同类错误输出统一格式，便于日志检索与用户理解。
- **UI 反馈闭环**：用户触发动作后必须有明确反馈（成功/失败/不可执行）。
- **注释解释 Why**：注释只保留设计意图与反直觉约束，不复述代码字面行为。

## 反模式档案 (Anti-Pattern Archive)
- **伪成功提示**：请求失败但仍 toast 成功。→ 先判定返回状态，再给成功反馈。
- **隐式失败吞噬**：`fetch` 未检查 `ok`，错误被当成功处理。→ 统一请求包装并抛出语义化错误。
- **功能可用性静默退化**：剪贴板/API 不可用时直接失败且无提示。→ 能力探测 + 回退路径 + 失败提示。
- **白名单安全模型**：通过硬编码"保护名单"防止误删。→ Manifest 追踪所有权，只操作自己管理的资源。
- **SDK 返回类型假设**：假定 SDK 返回值的数据结构（Array vs String、嵌套格式）而不验证 `typeof`。→ 用 `typeof` / `Array.isArray` 分支处理 + probe 脚本验证实际返回。
