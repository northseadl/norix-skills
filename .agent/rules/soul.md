# Soul — Agent 技术原则与设计哲学
> 跨项目通用的高维判断力。每条法则都必须通过“删掉它 Agent 会变蠢”的测试。

## 架构原则 (Architectural Principles)
- **Code is Truth**：文档是声明，运行结果才是事实；关键行为必须通过源码或 probe 验证。
- **100% 必须语义化**：涉及缩放场景时，“100%”应映射到业务语义基准（如最佳适配），而非裸物理倍率。
- **状态分层**：把“内容状态”“视图状态”“派生状态”分离，避免一个变量承载多种语义。
- **失败显式化**：I/O、网络、导出等边界操作必须校验返回值和状态码，失败必须可见。
- **缓存可控**：实时面板类系统默认 `no-store`，避免调试与刷新结果被缓存污染。
- **幂等优先**：刷新、保存、重试、导出等操作应可重复执行且结果可预测。
- **最小依赖面**：能用标准库解决的能力不引入额外依赖，降低维护熵。

## 代码哲学 (Code Philosophy)
- **Guard Clause First**：先处理空值、非法输入、前置条件失败，再进入主路径。
- **单一职责函数**：测量、缩放、下载、复制、请求解耦，函数名直接表达意图。
- **错误语义统一**：同类错误输出统一格式，便于日志检索与用户理解。
- **UI 反馈闭环**：用户触发动作后必须有明确反馈（成功/失败/不可执行）。
- **注释解释 Why**：注释只保留设计意图与反直觉约束，不复述代码字面行为。

## 反模式档案 (Anti-Pattern Archive)
- **伪成功提示**：请求失败但仍 toast 成功。
  - 替代：先判定返回状态，再给成功反馈。
- **隐式失败吞噬**：`fetch` 未检查 `ok`，错误被当成功处理。
  - 替代：统一 `requestJson` 包装并抛出语义化错误。
- **缩放耦合容器尺寸**：缩放基准与容器变化脱钩导致“100%”失真。
  - 替代：`baseScale`（适配基准）× `zoom`（用户倍率）双层模型。
- **功能可用性静默退化**：剪贴板/API 不可用时直接失败且无提示。
  - 替代：能力探测 + 回退路径 + 失败提示。
