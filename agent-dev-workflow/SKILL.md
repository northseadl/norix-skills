# Agent 全栈开发范式 (Portable Development Paradigm)

> **本文件是可独立迁移的完整开发范式包。**
> 部署到任意环境，Agent 即获得完整的工程方法论 + 可执行协议。
> 不依赖外部 user rules、不依赖特定项目配置，自身即为 Single Source of Truth。
>
> 触发方式：`//dev {协议名或自由描述}`。协议执行完即退出，不残留上下文。

# 第一部分：开发哲学 (Development Philosophy)

> Agent 在执行任何协议前，必须内化以下原则。这些原则是所有协议的判断根基。

## 1.1 架构原则

- **Code is Liability**：代码是负债。能用现有库解决的不写代码，能用配置解决的不写逻辑。追求 Negative Line of Code (NLOC)。
- **Forward-Only**：除非用户明确要求兼容旧版本，否则默认不考虑向后兼容性。采用当前最优技术栈与模式。
- **Monorepo First**：全栈项目强烈推荐统一仓库。Agent 可完整分析依赖链、跨端共享类型契约，实现原子交付。
- **Flat Over Nested**：目录结构扁平化优先，避免深层级联。文件系统必须反映业务逻辑关系。
- **Entropy Kills**：禁止 `common/utils/helpers/misc` 等低内聚垃圾桶目录。按高内聚功能领域拆分。
- **Idempotency**：所有脚本和接口必须幂等。
- **Statelessness**：服务尽量无状态，状态下沉至数据库或缓存。
- **Fail Fast**：让错误在编译期暴露，而不是运行期。所有边界必须有卫语句。

## 1.2 代码哲学

- **自解释命名**：变量/函数命名具备自解释性，消除对"What"注释的依赖。注释只解释 **Why** 和 **Warning**。
- **死代码零容忍**：未引用的变量、不可达分支、注释掉的代码块必须物理删除。Git 是历史工具，不是代码库。
- **注释忠实度**：修改逻辑时必须同步修改或删除相关注释。错误注释比没有注释更有害。
- **显式优于隐式**：使用显式类型，避免 `any`/`interface{}`。让类型系统承担尽可能多的约束。
- **Skeptical Empiricism**：不要完全信任文档。文档是"主张"，代码是"事实"。不确定时编写 probe 脚本验证。

## 1.3 迭代循环

无论多复杂的需求，必须坚守 **微循环** 节奏：

```
  ┌─────────┐    ┌──────┐    ┌───────────┐    ┌──────┐
  │ Develop  │ →  │ Test │ →  │ Refactor  │ →  │ Test │
  │ 让它跑起来│    │ 验证行为│    │ 清理结构  │    │ 回归验证│
  └─────────┘    └──────┘    └───────────┘    └──────┘
               ↑                                  │
               └──────────── 下一个微循环 ──────────┘
```

- **Develop**：先让代码能跑，哪怕结构有一丝不完美。
- **Test**：编写验证/测试。若环境未建立自动测试，用 CLI 手动 Probe 充当守卫。
- **Refactor**：在测试通过的安全网下清理结构。**此步骤坚决不增加任何新 Feature。**
- **Test**：回归验证未破坏。

## 1.4 反模式档案

| 反模式 | 正确做法 |
|---|---|
| 在 handler/controller 中直接操作数据库 | 业务逻辑 → service 层，数据操作 → store/repo 层 |
| `db.Save()` 全量更新 | `db.Model().Select("*").Updates()` 精确更新 |
| `_ = func()` 吞掉错误 | 记录日志或向上传播，绝不忽略 |
| `panic` 处理业务错误 | 返回 `error`，只在不可恢复的初始化场景 panic |
| Append-Only 配置/文档 | 定期压缩、替换、淘汰，维持信噪比 |
| 魔法数字/硬编码字符串 | 定义为命名常量，集中管理 |

# 第二部分：双脑架构 (Dual-Brain Architecture)

> Agent 的工程知识系统。在每个项目中维护两个知识文件，替代无序的沟通累积。

## 2.1 架构定义

| 大脑 | 路径 | 上限 | 性质 |
|---|---|---|---|
| **Agent Brain** | `.agent/rules/soul.md` | ≤200 行 | 跨项目通用的技术原则与设计哲学 |
| **Project Brain** | `.agent/rules/domain.md` | ≤300 行 | 当前项目的架构真相与领域知识 |

## 2.2 结构模板

### soul.md

```markdown
# Soul — Agent 技术原则与设计哲学
> 跨项目通用。每条法则必须通过"删掉它 Agent 会变蠢"测试。

## 架构原则
[通用系统设计法则：超时策略、状态管理、等待模型等]

## 代码哲学
[编码层面的通用真理：命名、注释、复杂度控制]

## 反模式档案
[已验证的错误模式 + 正确替代方案。不记录具体 Bug，只记录模式]
```

### domain.md

```markdown
# Domain — {项目名} 领域知识图谱
> 新开发者读完此文件即可独立工作。只描述当前真相，不记录历史变更。

## 系统架构
[技术栈、目录结构、核心模块关系]

## 核心数据流
[关键业务流程的数据流向 ASCII 图]

## 架构决策
[当前生效的关键决策，表格形式: 决策 | 说明 | 理由]

## 构建与运行
[Build / Test / Dev 命令]
```

## 2.3 准入标准

写入大脑的每条内容必须通过全部检查：

| 检查 | 说明 |
|---|---|
| **删除测试** | 删掉这条，Agent 判断质量会下降吗？ |
| **通用性** | soul: 跨项目通用？domain: 描述当前架构真相？ |
| **非冗余** | 与已有条目重复？能否合并？ |
| **非事件** | 是"应该怎么做"还是"发生了什么"？日记 → 拒绝 |
| **时效性** | (domain) 描述的还是当前状态？过期 → 删除 |

## 2.4 进化操作

> **绝对禁止 Append-Only。大脑不是日志文件。**

| 操作 | 触发条件 | 执行方式 |
|---|---|---|
| **Evict** | 条目不再通过准入标准 | 直接删除 |
| **Compress** | 多条相关条目可归纳为更高维法则 | 合并 N → 1 |
| **Replace** | 新认知覆盖旧认知 | 直接替换 |
| **Promote** | domain 条目验证为跨项目通用 | domain → soul |
| **Demote** | soul 条目实际只适用当前项目 | soul → domain |
| **Create** | 全新的、不可替代的知识 | 新增条目 |

## §1 项目初始化 (init)

**何时使用**：新仓库、接手已有仓库、或用户要求梳理项目结构。

### 执行步骤

```
1. SCAN — 扫描仓库结构
   - list_dir 根目录（depth=2）
   - 识别：语言/框架（package.json → Node, go.mod → Go, pyproject.toml → Python ...）
   - 识别：现有 Agent 配置（.agents/, .agent/, .claude/, AGENTS.md 等）
   - 检查是否已有 soul.md / domain.md

2. BLUEPRINT — 生成架构蓝图
   输出 Markdown 蓝图，覆盖：
   - 技术栈摘要（语言 + 框架 + 包管理）
   - 目录结构树（当前真实结构，非理想化模板）
   - 核心模块关系（如有多模块）
   - 构建/测试/运行命令

3. BRAIN INIT — 初始化双脑（仅当不存在时）
   - soul.md → 从第一部分开发哲学中提取种子内容（≤30行核心原则）
   - domain.md → 从步骤 2 的蓝图精炼（技术栈 + 目录 + 核心数据流）
   - 路径：使用仓库中已有的 Agent 配置目录；若无，创建 .agent/rules/

4. OUTPUT — 向用户展示蓝图 + 双脑初始内容，等待确认后写入
```

## §2 工作区演变 (evolve)

**何时使用**：一轮开发完成后需固化新认知，或用户感觉双脑文件质量退化。

### 执行步骤

```
1. READ — 通读双脑全文
   - view_file soul.md（完整）
   - view_file domain.md（完整）
   - 统计行数，计算占上限比例

2. ASSESS — 评估每条内容
   对每个条目执行 §2.3 准入标准的四项检查：
   ┌─────────────────────────────────────────────────┐
   │ 删除测试: 删掉它，Agent 判断力会下降吗？        │
   │ 时效性:   描述的还是当前真相吗？                │
   │ 非冗余:   与其他条目重复/可合并？                │
   │ 非事件:   是"应该怎么做"还是"发生了什么"？      │
   └─────────────────────────────────────────────────┘

3. OPERATE — 按 §2.4 执行进化操作
   优先级：Evict > Compress > Replace > Promote/Demote > Create

4. WRITE — 执行文件修改
   - 如果行数 > 上限 80%，先压缩再新增
   - 修改后文件必须仍符合 §2.2 结构模板
   - 输出 diff 摘要告知用户变更内容

5. VERIFY — 行数验证
   - soul.md ≤ 200 行
   - domain.md ≤ 300 行
   - 若超限，强制再次压缩
```

## §3 安全审计 (audit)

**何时使用**：提交代码前、CI 流程中、或周期性安全检查。

### 执行步骤

```
1. SCAN — 使用 grep_search 扫描高危模式
   扫描目标：所有已 tracked 的文件（排除 .git/、node_modules/、vendor/）

   扫描模式清单（正则）：
   ┌───────────────────────────────────────────────┐
   │ API Key:     ['"](?:sk|ak|api)[_-][\w]{20,}   │
   │ AWS:         AKIA[0-9A-Z]{16}                  │
   │ Private Key: -----BEGIN (?:RSA |EC )?PRIVATE   │
   │ Password:    password\s*[:=]\s*['"][^'"]{8,}   │
   │ Token:       token\s*[:=]\s*['"][^'"]{20,}     │
   │ JWT:         eyJ[A-Za-z0-9_-]{10,}\.eyJ        │
   │ Connection:  ://\w+:\w+@[\w.-]+               │
   └───────────────────────────────────────────────┘

2. FILTER — 排除虚构值
   以下不报告：
   - 包含 test / example / dummy / changeme / placeholder / xxx / your-
   - 在 .env.example / .env.sample 中的值
   - 文档文件（.md）中的说明性示例

3. REPORT — 输出审计报告
   | 文件 | 行号 | 类型 | 严重度 | 片段 |
   |---|---|---|---|---|

   严重度分级：
   - 🔴 CRITICAL: 真实 API Key / Private Key / Connection String
   - 🟡 WARNING:  疑似密码或 Token（可能是占位符）
   - 🟢 INFO:     模式匹配但大概率无害

4. FIX — 提供修复建议
   - 每个 CRITICAL 项建议使用环境变量替代
   - .gitignore 补充建议（如缺少 .env）
```

## §4 Git Hooks 注入 (hooks)

**何时使用**：新仓库需要建立提交守卫，或现有 hooks 需要更新。

### 执行步骤

```
1. DETECT — 检查现有 hooks
   - ls .git/hooks/ 查看已有 hook
   - 如果已存在同名 hook，提示用户确认覆盖

2. GENERATE — 根据仓库语言生成适配的 hook

   pre-commit hook（安全扫描）：
   - 扫描 staged 文件中的硬编码凭据（模式同 §3）
   - 检查是否有超大文件（>5MB）被意外 staged
   - 检查是否有 .env / *.pem 被意外 staged

   pre-push hook（分支保护）：
   - 拦截直接 push 到 main/master
   - 仅允许从 develop/feature/ 通过 merge 方式进入
   - 验证 commit message 格式（如已定义规范）

3. INSTALL — 写入并设置权限
   - 写入 .git/hooks/{hook_name}
   - chmod +x
   - 验证 hook 可执行

4. VERIFY — 执行空运行确认 hook 工作
```

## §5 代码健康度检查 (health)

**何时使用**：重构前评估、代码审查、或定期维护。

### 执行步骤

```
1. DEAD CODE — 扫描未引用的导出
   策略：grep 导出符号 → 反向搜索引用 → 标记零引用项
   目标：导出函数、导出常量、导出类型

2. ZOMBIE COMMENTS — 扫描注释掉的代码块
   模式：连续 3+ 行注释中包含代码语法特征
   （如 //, if, return, func, const, import 等）

3. DIRECTORY ENTROPY — 目录结构评估
   检查项：
   - 是否存在 utils/ common/ helpers/ misc/ 等低内聚目录
   - 单目录文件数是否超过 15 个（扁平化阈值）
   - 嵌套深度是否超过 5 层

4. REPORT — 输出健康度报告
   格式：Markdown 表格 + 总分（满分 100）
   扣分规则：
   - 每个死代码导出 -2
   - 每段僵尸注释 -1
   - 每个低内聚目录 -5
   - 嵌套深度超标 -3/层
```

# 第四部分：运行时规则

## 智能路由

当用户使用 `//dev {自由描述}` 且描述不严格匹配协议名时：

```
IF 描述包含 "新项目" / "初始化" / "架构" / "结构"  → 路由到 §1 init
IF 描述包含 "演变" / "进化" / "大脑" / "brain"     → 路由到 §2 evolve
IF 描述包含 "安全" / "审计" / "密钥" / "泄露"      → 路由到 §3 audit
IF 描述包含 "hook" / "钩子" / "提交守卫"           → 路由到 §4 hooks
IF 描述包含 "健康" / "死代码" / "重构评估" / "熵"   → 路由到 §5 health
ELSE → 展示协议索引表，请用户明确选择
```

## 隔离约束

本技能的所有协议执行完毕后，Agent 必须遵守：

1. **不保留协议上下文**——不将本技能的执行细节带入后续对话
2. **不自动触发**——即使检测到仓库缺少 soul.md，也不主动建议运行 init
3. **不篡改宿主规则**——双脑修改仅限项目目录内的文件，不触碰 user global rules
4. **不复制粘贴本文件**——如需在其他 Agent 配置中引用本范式，只注入加载指令而非内容副本
