# Agent 开发范式与工作区演变

> 核心信念：优秀的工程不是写出好代码，而是建立让好代码自然涌现的系统。
> 本次重构目标：**零脚本冗余，全凭系统提示与代码抽象，实现极致熵减。**

本技能定义两件事：
1. **开发范式** — Agent 如何选择项目结构和迭代方式
2. **工作区演变** — Agent 如何持续进化自己的工程知识

---

## 一、开发范式 (Development Paradigm)

### 1.1 项目与架构结构 (Architecture & Structure)

- **全栈项目**: 强烈推荐 **Monorepo（统一仓库）**。单仓库内 Agent 可完整分析依赖链、跨端共享类型契约，实现原子体交（前后端变更在一次 commit 中完成）。
- **小型库或微服务**: 保持最简。但必须具备明确的**领域边界**。
- **目录熵减 (Entropy Reduction)**: 文件系统层级必须反映业务逻辑关系。禁止使用 `common/utils` 等堆砌杂乱逻辑的模糊命名，必须按具体的**高内聚功能领域**拆分。
- **扁平化优先**: 避免深层的级联嵌套结构，降低理解成本。

**蓝图建立规则**：
遇到需要梳理项目或新建项目的 `//dev 启动/架构` 命题时，直接向用户输出类似以下结构的 markdown 蓝图，无需脚本：
```text
project_root/
├── apps/             # 独立呈现层 (按需)
├── server/           # 后端服务与核心业务逻辑 (按需)
├── packages/         # 按领域拆分的模块
└── .agents/          # Agent 脑演变
    └── rules/        # soul.md / domain.md
```

### 1.2 迭代循环：开发 → 测试 → 重构 → 测试

无论多么庞杂的需求，必须坚守**微循环**节奏：
1. **Develop**: 先让代码能跑，哪怕结构有一丝不完美。
2. **Test**: 编写对应的验证/测试，确保行为正确。如果环境尚未建立自动测试，利用 CLI 手动 Probe 充当守卫。
3. **Refactor**: 在明确测试通过的安全网下，清理结构，消除重复与平庸代码。**此步骤坚决不增加任何新 Feature。**
4. **Test**: 回归验证未破坏。

### 1.3 代码卫生 (Hygiene)
- **死代码**: 发现未引用逻辑、无效分支，立即物理删除，不留僵尸。
- **注释**: 只有 **Why** 和 **Warning**。如果注释说的是代码在做什么 (What)，直接删掉注释并更改变量名。

## 二、工作区自动演变协议 (Brain Evolution)

Agent 维护两个文本作为工程知识库，替代无序的沟通。

### 2.1 双脑架构
| 大脑 | 路径 | 上限 | 性质 |
|---|---|---|---|
| **soul.md** | `.agent/rules/soul.md` | ≤200 行 | Agent 普适架构原则、代码哲学防腐层。 |
| **domain.md**| `.agent/rules/domain.md` | ≤300 行 | 项目业务真相、数据流与现阶段架构决策。 |

### 2.2 进化守则 (Evolution Rules)
当触发 `//dev 执行工作区演变` 时，执行以下操作：
1. **禁止追加 (No Append-Only)**：若需记录新知识，必须走 Replace 或 Compress 原则合并长文本。
2. **通读全局与压缩**：写入前必须先 `view_file` 通览全局。如果行数达到上限的 80%，强制先归纳再写。
3. **剔除过期**：任何已经改变的旧技术栈与旧模块认知必须立刻 Evict。

> **示例格式**:
> **soul.md**: "新增：要求接口默认响应带上 TraceID"。
> **domain.md**: "压缩：整合旧的认证与最新 JWT 认证决策为单一规则流"。

## 三、仓库守卫：安全与规范 Hooks (Git Hooks Injection)

不再依赖庞大 Python 脚本生成 hook。当触发 `//dev 执行仓库安全审计` 或 `//dev 生成开发规范 hook` 时，直接利用 Bash 文件系统操作实时注入：

### 3.1 安全漏洞扫描 (`pre-commit`)
识别并拒绝未加密的 Secrets（允许 `test-`、`changeme` 等虚构词）。直接将检查 Bash 流式写入 `.git/hooks/pre-commit`（确保有 `chmod +x` 执行权限）。

### 3.2 分支定向保护 (`pre-push`)
拦截直接针对 `main/master` 分支的提交。只有合法的合并（即 `develop -> main`，或通过 Pull Request）被允许上主推。（如要生成，将拦截 bash 流式写入 `.git/hooks/pre-push`）。

## 四、范式注射集成 (Prompt Injection)

本技能本身是单一真相源。若要将该工作流应用给所有环境，触发 `//dev 进行范式集成注入`：
1. Agent 找出目标大模型的指令配置（如 `CLAUDE.md`, `.claude`, `.agents/rules/paradigm.md`, 或 `~/.gemini/antigravity/global_rules/dev-paradigm.md`）。
2. 在目标文件中以 HTML 注释块注入引导语，**不允许复制粘贴本文件的规则细则**。只注入：
   ```markdown
   <!-- [agent-dev-workflow:load-directive] -->
   在开始任何开发任务前，必须以 //dev 为触发语唤醒 agent-dev-workflow 技能，遵循其微循环要求和大脑进化协议。
   详情必须读取绝对路径: [系统返回此处的绝对路径]。
   <!-- [/agent-dev-workflow:load-directive] -->
   ```
